# Configuration

meeting-hive reads its config from an OS-appropriate location:

- **macOS / Linux**: `~/.config/meeting-hive/config.yaml` (or `$XDG_CONFIG_HOME/meeting-hive/config.yaml`)
- **Windows**: `%APPDATA%\meeting-hive\config.yaml`

The file is generated by `meeting-hive init` (the macOS installer runs it for you with your answers). On Linux and Windows, run `meeting-hive init` manually after installing the package.

Changes take effect on the next run.

## Top-level shape

```yaml
source:
  adapter: <name>         # which source adapter to use
  config: { ... }         # per-adapter config, opaque

vocabulary:
  adapter: <name>         # which vocabulary adapter to use
  config: { ... }

summarizer:
  adapter: <name>         # REQUIRED. No default — pick explicitly.
  config: { ... }

# Classification rules (evaluated in order; first match wins).
scope: <string>            # default: "work"
internal_domains: [ ... ]
title_patterns:  [ ... ]
internal_only:   { ... }
email_rules:     { ... }
domain_rules:    { ... }
```

## Source adapters

### `granola`

Reads from [Granola](https://granola.ai)'s local cache, with REST API fallback. Works on macOS; will work on Windows when Granola ships a Windows client (point `cache_path` at the Roaming/AppData location via config).

```yaml
source:
  adapter: granola
  config:
    # All optional — sensible defaults per OS.
    cache_path: ~/Library/Application Support/Granola
    cache_filename: cache-v6.json
```

### `fathom`

[Fathom](https://fathom.ai) via its [public REST API](https://developers.fathom.ai/). Cross-platform, no local app required.

```yaml
source:
  adapter: fathom
  config:
    # Optional — defaults shown.
    base_url: https://api.fathom.ai/external/v1
    api_key_env: FATHOM_API_KEY     # env var sourced from secrets.env
    retries: 3
```

Put your key in `secrets.env`:

```
FATHOM_API_KEY=fthm_...
```

Get a key from your Fathom account settings. The adapter uses cursor pagination to pull all meetings created in the lookback window and fetches transcripts on demand per meeting.

### `markdown`

Generic, tool-agnostic: reads meetings from a directory of YAML-frontmatter markdown files. Use this with any tool that exports markdown (Meetily, Hyprnote, OpenWhispr, Fathom export, hand-written notes, …).

```yaml
source:
  adapter: markdown
  config:
    path: ~/Documents/meetings          # required
    pattern: "**/*.md"                   # optional (default)
    id_field: id                         # optional; falls back to filename stem
    attendees_field: attendees           # optional; tries `attendees` then `invitees`
    ignore_summary: true                 # optional; skip files with `type: meeting-summary`
```

Expected frontmatter on each meeting file:

```yaml
---
title: "Kickoff with Acme"
date: 2026-04-17
time: "15:00-15:30"           # optional
attendees:                     # or `invitees:`
  - alice@acme.com
  - me@example.com
id: optional-stable-id         # optional
---
(transcript body — everything below the frontmatter)
```

## Vocabulary adapters

### `sqlite` (recommended default)

A local SQLite database managed by meeting-hive itself. Cross-platform (macOS / Linux / Windows). Mutable via `meeting-hive vocab add/remove/import/export` CLI.

```yaml
vocabulary:
  adapter: sqlite
  config:
    # Optional — defaults to the platform data dir
    # (macOS/Linux: ~/.local/share/meeting-hive/vocabulary.db,
    #  Windows: %LOCALAPPDATA%\meeting-hive\vocabulary.db).
    db_path: ~/.local/share/meeting-hive/vocabulary.db
```

Populate it:

```bash
meeting-hive vocab add "Post Grass" "Postgres"
meeting-hive vocab import yaml my_vocab.yaml
meeting-hive vocab import wispr                 # one-time seed from Wispr Flow
```

### `wispr`

Read-only wrapper over the [Wispr Flow](https://wisprflow.ai) dictation app's personal vocabulary (SQLite). macOS and Windows — [Wispr Flow has no native Linux](https://docs.wisprflow.ai/articles/1036674442-supported-devices-and-system-requirements).

Use this if you already maintain a vocabulary inside Wispr and want meeting-hive to read from there (no duplication, no migration).

```yaml
vocabulary:
  adapter: wispr
  config:
    # Optional — defaults to the platform-standard Wispr location.
    db_path: ~/Library/Application Support/Wispr Flow/flow.sqlite
```

CLI mutations (`add`, `remove`, etc.) are **not supported** — edit entries in the Wispr app. The CLI will print a clear error telling you where to edit.

### Writing your own adapter

Drop a file in `meeting_hive/vocabs/` that exposes a class implementing `VocabularySource` (a `load() -> dict[str, str]` method) or `MutableVocabularySource` (adds `add`/`remove`/`clear`). Register in the `_BUILTINS` dict. Same pattern for source adapters in `meeting_hive/sources/`.

## Summarizer adapters

No default — you must pick one. The macOS installer writes it during setup; edit `summarizer.adapter` by hand to switch later.

The prompt template is shared across adapters (defined in `meeting_hive/summarizers/__init__.py`) so the output shape stays consistent regardless of which backend you use.

### `anthropic`

Claude family via the Anthropic API.

```yaml
summarizer:
  adapter: anthropic
  config:
    model: claude-sonnet-4-5-20251001   # default
    max_tokens: 2000                     # default
    api_key_env: ANTHROPIC_API_KEY       # default; env var sourced from secrets.env
    base_url: https://api.anthropic.com  # optional override (LiteLLM proxy, Bedrock, etc.)
    retries: 3                           # default
```

Get a key at [console.anthropic.com/settings/keys](https://console.anthropic.com/settings/keys).

### `openai`

GPT / o-series via OpenAI's Chat Completions API.

```yaml
summarizer:
  adapter: openai
  config:
    model: gpt-4o                        # default; any OpenAI model ID works
    max_tokens: 2000
    api_key_env: OPENAI_API_KEY
    base_url: https://api.openai.com/v1  # optional override (Azure, proxy, etc.)
    retries: 3
```

Get a key at [platform.openai.com/api-keys](https://platform.openai.com/api-keys).

### `ollama`

Local LLM via an [Ollama](https://ollama.com) server. No API key. Requires the Ollama daemon running and the chosen model already pulled (`ollama pull llama3`).

```yaml
summarizer:
  adapter: ollama
  config:
    base_url: http://localhost:11434     # default
    model: llama3                        # default
    timeout: 300                         # default; local models can be slow
    retries: 3
```

If the server isn't reachable when the pipeline runs, the job fails loudly with a `SummarizerUnavailable` error and a notification.

### Using OpenAI-compatible servers

Both `openai` and `anthropic` adapters accept a `base_url` override, which means the same adapter code talks to any server that speaks the respective wire protocol. This covers most self-hosted and proxied LLM setups without needing a new adapter.

**OpenAI wire format** — use the `openai` adapter and point `base_url` at:

| Server | `base_url` | Notes |
|---|---|---|
| [LM Studio](https://lmstudio.ai/) | `http://localhost:1234/v1` | Desktop app. Serves any GGUF model. |
| [llama.cpp](https://github.com/ggml-org/llama.cpp) (`llama-server`) | `http://localhost:8080/v1` | CLI server. Very lightweight. |
| [Jan.ai](https://jan.ai/) | `http://localhost:1337/v1` | Desktop app. Cross-platform. |
| [vLLM](https://docs.vllm.ai/) | `http://localhost:8000/v1` | Production-grade serving. |
| [LocalAI](https://localai.io/) | `http://localhost:8080/v1` | Drop-in OpenAI replacement. |
| [OpenRouter](https://openrouter.ai/) | `https://openrouter.ai/api/v1` | Proxy to 100+ models (Claude, Gemini, Mistral, Llama, …). |
| [LiteLLM](https://docs.litellm.ai/) proxy | `http://localhost:4000/v1` | Routes to any provider. |

Example:

```yaml
summarizer:
  adapter: openai
  config:
    base_url: http://localhost:1234/v1
    model: qwen2.5-7b-instruct
    api_key_env: OPENAI_API_KEY   # most local servers ignore the value, but the SDK requires one to be set
```

For local servers that don't check auth, set `OPENAI_API_KEY=local` (or anything non-empty) in `secrets.env` to satisfy the SDK.

**Anthropic wire format** — use the `anthropic` adapter with `base_url`. Commonly used with LiteLLM proxy or AWS Bedrock via a compat layer.

### Writing your own summarizer adapter

File in `meeting_hive/summarizers/` with a class implementing `Summarizer` (one method: `summarize(transcript, title, attendees) -> str`). Register in `_BUILTINS`. Use `format_prompt()` and `strip_fences()` from `summarizers/__init__.py` so your output is shape-compatible with the pipeline.

## Classification rules

Rules are evaluated in order. The first match wins.

### 1. `title_patterns` — regex against meeting title

Use for **named recurring meetings** or **entities whose name appears in the title** but whose domain is unknown/inconsistent.

```yaml
title_patterns:
  - match: "(?i)founders friday"
    type: internal
    entity: founders

  - match: "(?i)\\bacme\\b"
    type: client
    entity: Acme
```

Regex flags are inline (`(?i)` for case-insensitive). Use `\\b` for word boundaries (YAML escapes the backslash).

Fires BEFORE any domain check, so this is where you pin the canonical category of a specific recurring meeting even when attendees change week to week.

### 2. `internal_only` — every attendee is internal

Fires when 100% of attendees match `internal_domains`.

```yaml
internal_only:
  default_subtype: alignment
  title_hints:
    - match: "(?i)founders"
      subtype: founders
    - match: "(?i)roadmap"
      subtype: roadmap
    - match: "(?i)interview"
      subtype: interview
```

The `internal:` category then gets a subtype from the first `title_hints` entry that matches. If nothing matches, uses `default_subtype`.

Common subtypes you might use: `founders`, `alignment`, `roadmap`, `product`, `sales`, `interview`, `1-on-1`, `learning`. You can invent your own — they're just string labels.

### 3. `email_rules` — exact email match

Use for individual contacts whose domain is too generic (gmail, outlook.com) to use `domain_rules`.

```yaml
email_rules:
  jane.doe@gmail.com:
    type: vendor
    entity: Jane Doe (Freelance Designer)

  partner@hotmail.com:
    type: client
    entity: Partner Co
```

### 4. `domain_rules` — domain of any external attendee

The bulk of classifications. When any attendee's email domain matches, the meeting is classified.

```yaml
domain_rules:
  acme.com:
    type: client
    entity: Acme

  stripe.com:
    type: vendor
    entity: Stripe

  ycombinator.com:
    type: investor
    entity: Y Combinator
```

**Important**: the first matching domain wins. If a meeting has both `@acme.com` and `@stripe.com` attendees and both are in the rules, the order in which meeting-hive iterates (roughly: invite order) decides. If that matters, promote the more-specific one to `title_patterns`.

## Allowed category types

| Type | When to use |
|---|---|
| `client` | Paying customer or active prospect under contract |
| `vendor` | Supplier, freelancer, tool-we-pay-for, agency, consultant |
| `internal` | Your own team. Value is the subtype (`alignment`, `founders`, etc.) |
| `investor` | VC, angel, LP |
| `community` | Programs, associations, external communities you participate in |
| `event` | One-off events not covered by the above |

You can invent new types — meeting-hive doesn't enforce a fixed enum. But stick to a small stable set for grep-friendliness.

## UNCLASSIFIED

When no rule matches, meeting-hive logs a snippet like:

```
UNCLASSIFIED: "Conversation with New Prospect"
  attendees: jane@newprospect.xyz, me@mycompany.com
  # Suggested domain_rules entry(ies):
  #   newprospect.xyz:
  #     type: ???   # client | vendor | investor | community
  #     entity: "???"
```

Paste the suggestion into `domain_rules`, fill in `type` + `entity` with canonical values, save. The next run (or a manual one) picks up the meeting.

No file is written. No half-state in the archive.

## Config validation

If `config.yaml` is malformed YAML, meeting-hive **does not** run. Instead it sends:

```
❌ config.yaml invalid (line 47)
```

Fix the syntax error, and the next run proceeds normally.

## Tips for keeping the config clean

- **One canonical name per entity.** "Acme" not "Acme Corp" sometimes and "Acme" other times. Pick one, stick with it.
- **Title patterns for recurring meetings, domain rules for everything else.** Don't use title patterns as a catch-all — they become fragile when meeting titles change.
- **When a domain covers multiple entities (holding companies, agencies)** → use `title_patterns` to distinguish.
- **When an entity has a sub-brand** (e.g., "Acme" globally, "Acme Pro" is a product line) → keep them as one `entity`. Use the slug in the filename or the title in the frontmatter to preserve the sub-brand context.
- **Review UNCLASSIFIED entries periodically** — they're often new opportunities your system doesn't yet know about.
